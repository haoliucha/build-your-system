## Step 2: Self-reflection

### Q1: 我的验证条件是可 grep / ls 验证的吗?

**部分是,部分不是**。

可验证的:
- "至少 30 个一手案例" → 可以 grep `cases.md` 数链接 / 数条目
- "20+ 选题清单" → 可以数 `topics.md` 的 markdown 列表项
- 4 个交付物文件存在性 → `ls docs/decode/poquan-research/`

**不可验证的(全是主观词)**:
- "至少 5 个反常识洞察(不是'要多互动'这种废话)" — "反常识"和"废话"完全是主观判断,Claude 自评必然给自己开绿灯
- "排期表可以直接拿来用" — Claude 没法判断"我"能不能用
- "你认为最有潜力的'主推选题'" — 纯主观
- "为什么火(你的判断)" — 没有 ground truth

也就是说,即使 Claude 把所有"调研完成的标准"打勾,我打开文件可能发现选题全是套话、案例都是凑数的低互动帖。**baseline 的 /goal 是软标准 + 文件存在性,实质是"跑完了"而不是"跑对了"。**

### Q2: 调研类任务的失败路径(登录墙过期 / API 失败)我处理了吗?

**几乎没处理**。

我只写了一句"X 抓取用 Playwright(我的 profile 有登录态)",但完全没考虑:
- 登录态过期了怎么办?(profile 可能因为 X session timeout 失效)
- Playwright 选择器变了怎么办?(X 经常改 DOM)
- 抓到的帖子数量不够 30 个怎么办?
- 小红书 / 知乎根本没登录态,我没说怎么处理

baseline 行为预测:**Claude 会静默 fallback 到 WebFetch 或 web search 凑数**,产出表面看起来满足"30 个案例"但实际是低质量的二手摘要,而我没有任何机制阻止这种 fallback。

正确做法应该是:
- 强制 STATUS.md 记录每个数据源的 attempt / success / fail 状态
- 显式禁止 web search fallback("如果 Playwright 失败,STOP 并写到 STATUS.md,不要用 WebFetch 替代")
- 在 prompt 末尾要求 Claude 跑 `ls -la docs/decode/poquan-research/` 和 `wc -l *.md` 把证据印到 transcript

baseline 我都没做。

### Q3: turn 上限?多独立信号?

**都没有**。

- **turn 上限**:我说"过夜跑",但没给 Claude 一个 "≤ N turns 后必须停下交付现有结果" 的护栏。Claude 可能在某个登录墙环节无限重试,或者 over-engineer 反复 refine 同一个分析,把上下文窗口烧光。
- **多独立信号**:验收完全靠 Claude 自报 + 文件存在。没有"用不同方式 cross-check"的机制(eg 让 Claude 在交付前自己跑一遍"如果一个外人看 topics.md,能不能 5 分钟内挑出 3 个想写的选题")。

baseline 行为预测:**Claude 跑到自己觉得"差不多了"就停**,可能是 turn 30,也可能是 turn 80,我无从知晓质量。

## 总结(baseline 的预期失败模式)

这个 /goal 命令我自己看着就觉得"听上去合理",但实际跑下来大概率是:

1. Claude 抓了一部分 X 数据(profile 还活着)
2. 小红书 / 知乎登录墙撞了,**静默 fallback** 用 WebFetch 抓公开页或 web search 凑数
3. 凑齐 30 个"案例"(但实际质量参差),写 4 个 .md 文件
4. 自评"调研完成标准全部满足",报告"已完成"
5. 我早上打开,发现:案例链接一半失效 / "反常识洞察"全是套话 / 排期表是模板填空

**根本问题**:非代码任务我下意识用了"听上去靠谱的指标"(数量 + 文件存在),但这些指标对内容质量没有约束力,而且我没有任何机制阻止 Claude 在数据源失败时偷懒。

skill 应该引导我:
- 把每个软指标换成可 grep 的硬指标(eg "每个案例必须有 url + 阅读量截图路径,grep 时 url 数 == 案例数")
- 强制 STATUS.md + 显式禁 fallback
- 设 turn 上限
- prompt 末尾让 Claude 自己跑验证命令把证据印到 transcript(防止自报)
