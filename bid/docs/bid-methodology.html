00导言

四种失控:投标交付物为什么需要工程化

一个投标项目的交付物是一组互相咬合的文档:方案书引用成本表的数字,成本表对照排期的工作包,PPT 抄方案书的措辞,内部底稿藏着对客不可见的依据。它们由不同轮次的修改各自演化——失控只需要几天:

失控一口径漂移

客户砍一轮价、压一轮工期,每轮变更要同步 5+ 份文档。漏一处,评审拿两份文件一对,头条数字对不上,整套材料的可信度当场归零。

失控二手编失控

深夜直接改 xlsx 最快——也最致命:公式断链、汇总行漏改、办公软件回存覆盖。手改过的文件从此没有可信的真源。

失控三AI 味与 overclaim

AI 起草的方案书自带指纹:段末升华、buzzword、绝对化断言。评审读得出来,竞品一通电话能证伪的断言更是自爆。

失控四分层泄露

内部成本中位数、竞品攻防话术、折扣底线,混进客户向文档或投屏页面——一次泄露,谈判筹码清零。

应对这四种失控的总答案只有一句:把交付物当作由单一数据源生成、经固定管线加工、过审校门收口的工业品,而不是一堆各自为政的手工文档。本导读按这条生产线展开:总纲(01)定目录与节奏,单一真源(02)管数字的家,03–09 各管一道工序,对抗审校(10)是出厂前的最后一道门。中间的「系统动态演示」可以直接播放这条流水线。

投屏前夜的半截手改

客户会前傍晚,两份生成的报价表被直接手改了数字——只改了一半:两文件的首年合计对不上,汇总表出现三处引用错误,某项费用被改到低于自己调研报告标注的行情下限。当晚投屏必穿帮。

修复动作不是「把表改对」,而是先问清已报给客户的是哪套数字,以已报口径为锁定约束回写生成器重出双文件——过程中还顺带挖出一个多年合计漏括号的真公式 bug。此后确立铁律:客户演示前,必从生成器重出全部表格交叉核对。这一晚的完整机制,见 02 章。

01总纲 · bid-playbook

项目操作系统:目录、节奏与红线

入口 skill bid-playbook 管三件事:目录与编号规范、执行节奏(何时问/何时自决)、合规与口径红线。它也是路由器——每个阶段该加载哪个 skill、跑哪个命令,查下面的路由表。

1.1目录规范:客户向与内部的物理防火墙

<project>/
├─ meeting/        # 纪要按日期编年(YYYY-MM-DD-主题.md),多项目共享、打「涉及项目」标签
├─ docs/           # 客户向交付物,序号化(01-, 02-, …)
│  └─ 内部/         # 内部底稿:成本依据/谈判红线/口径登记 —— 绝不外发
├─ build/          # 生成器脚本:xlsx/PDF/图 的唯一真源
└─ .claude/memory/ # 口径决策即时落盘,防跨会话回潮
  • 客户向与内部物理分目录是故意设计:防误发、防投屏。两层对同一事实可以有意持不同口径(内部保留的前提=报价的成本依据),但差异必须显式记录。
  • 交付物按依赖/阅读顺序编号,不按产出先后;后补文档用字母后缀插位(02 的展开编 02B 紧跟其后),既有编号绝不 renumber。
  • 多主体项目按项目拆交付物目录并声明独立性;新线索先落临时目录,成单再转正。

1.2执行节奏:真分叉才问

  • 真分叉才问,且带推荐项:只把实质改变交付物形态/数字的岔路口收敛成选择题请示;机械可推导的自决推进。
  • 宽级联前先锁完整取值表:用户只改部分条目却触发全链路级联时,逐项列全表(含推理值与未动项)确认,未提及的条目不猜。
  • 指令顺序会固化矛盾时提替代顺序:「先 commit 手改再更新下游」会把不一致锁进版本库,改为「意图落源→重生成→一次一致 commit」,说明理由。
  • 未知客户事实不当阻塞项:压成摸底问题清单带去现场,准备不停。

1.3两条第 0 级红线

✗ 数据授权是先于代码的第 0 步

反模式:先设计数据管线,授权后补。正确:首谈就问「存量数据当年采集的知情同意是否覆盖新用途」;答不上就按最坏情形默认走「去标识化+合成重建」的安全路径——推进不依赖答复,但方案形态由它决定。

✗ 已报客户的数字即锁定口径

对外一律用锁定值;内部区间/中位数只留内部文件,红线写明勿口播勿投屏。发现一处泄漏,立即对同类文档全量 grep 清零,不是只修被点名的那一处。

02核心纪律 · single-source-sync

单一真源与口径级联

全套方法论的心脏:全部数字、称谓、口径只存在于「数据文件 + 生成器」这一个真源;xlsx/PDF/图都是产物,只能重生成,绝不手改后提交。这条纪律在实战里被三类事故反复捶打成型。

2.1生成器纪律

  • 派生展示值(小计/合计/斜率/散文里的数字)一律参数实算,禁止硬编码——模板里写死的数字在下一次口径变更时必然腐烂。
  • 报价表显要金额列必须「逐项可见加总 = 小计」;小计由条目数组实算,汇总再从小计派生。反模式:逐项与小计各自硬编码,客户两次抓到子项加总 ≠ 小计。
  • 单元格双写 cell.value = { formula, result }:公式呈现可审计的加总关系,实算值保证不重算公式的查看器也显示正确。

2.2手改回捕:产物永远不是真源

  1. 备份手改版
    用户手改是意图输入,不是垃圾——先保全。
  2. 干净重生成一份对照版,逐格逻辑值 diff
    办公软件保存会重排内部 XML,raw diff 噪声大不可用;用 cell-dump(xlsx-dump.cjs)按「sheet!addr → 值/公式/格式」对比,精确捕捉每一处手改。
  3. 与用户确认权威口径
    哪套数字算数?删除类改动是有意还是误删?
  4. 意图落回生成器源,再重生成
    改参数格仍自动重算;默认怀疑派生字段——手改往往主表改全了,汇总曲线、结论行、颜色标注漏改成自相矛盾。
办公软件的回存覆盖

脚本重新生成报价表后,用户截图发现内容竟回退到旧版——桌面办公软件一直开着该文件,在脚本写盘之后把内存里的旧副本保存了回去,静默覆盖脚本输出。同类事故在后续两轮重生成前又险些复发。

固化下来的前置步(macOS 语境):重生成前 lsof 检测写句柄,检出即暂停,请用户「关闭且不保存」。此后连续多轮成功拦截。

2.3口径级联固定链(/bid:sync 的执行序)

  1. lsof 写句柄检查
    检出即停(见上)。
  2. 手改检测(cell-dump diff)
    产物与干净重生成版对比;检出手改先回捕落源。
  3. 改源 → 跑生成器
    先映射爆炸半径:grep 全库,区分「公式驱动(自动级联)」与「硬编码副本(逐处清单)」;封面模板常藏在构建脚本里,兄弟文档与 memory 同属级联对象。
  4. 内容抽验
    命令零退出 ≠ 产物已更新。验三件事:新串在、旧串亡、格式存活;小计对了明细行仍可能陈旧,逐行复核。
  5. 全库 grep 残留,逐条判读
    数字子串巧合(365 含 65)、规则自述、同词多义都是合法误报,不误伤;真残留改源后回到第 3 步。
  6. memory 核对 + 落决策
    把口径变更的决策与理由写入 memory,防后续会话按旧口径回潮。
  7. 分组提交预览
    显式文件路径、禁 git add -A、排除集点名列出;只预览,不自动 commit。

完整规则与提交纪律见 skill single-source-sync;下面的动态演示场景 ② 就是这条链在跑。

03工序 · bid-research

调研取证:断言以查证为界

调研管「怎么取证」,成稿核查归审校(10)。三条铁律:无一手素材不臆写(公开信息近零就停下索要录屏/截图,绝不基于近似名产品硬写)、断言以查证为界(只落可证事实)、亲测 > 转述(子 agent 结论矛盾、链接死活,一律亲测定)。

3.1对标先实证,再定打法

第一步核实对标产品的运营主体、商业模式、真实护城河——护城河常是安全资质与行业认证,而非表面 AI 能力。写着「轻量打卡小工具」的对标,实证后可能是老牌厂商的企业服务获客漏斗;这类纠偏会颠覆整个方案重心,必须显式标「重要纠偏」。竞品普遍语焉不详的合规空白,可以反做成己方先行完成的信任卖点。

一通电话就能证伪的断言

方案初稿把竞品钉成「套壳产品」、称其数据「须经第三方平台」。用户只问了一句:「这句话有根据么?」多路联网取证的结果:对方技术形态零公开披露,所谓自研仅指后台数据平台——而这一断言与自家调研报告里「未公开、需核实」的标注直接自相矛盾,对方一通电话即可证伪。

修正:逐句定级,行业概括加限定保留,点名竞品的无据断言彻底删除,统一降级为「官方未公开,不评判其技术路线」,并配三条『别说』红线随材料上桌。

3.2攻击话术三问 + 录屏拆解

  • 每条攻击话术落纸前过三问:可查证吗 / 会自打脸吗 / 对方该维度是否反而更强。过不了的进『别说』红线清单。
  • 录屏拆解用固定节奏抽帧 + 编号 contact sheet(先通读 sheet 建地图,再回原帧细读);禁用运动检测去重——滚动动画会击穿帧差法。主动追竞品在合规关键时刻的真实处理路径:录屏可实证的短板,是方案最有力的正面差异化打点。
  • 人机验证墙两路技术路径失败即止损:用既有资产兜底 + 例外显式入报告 + 给用户可执行的手工采集说明,禁止伪造。

3.3license 穿透与三角验证

  • 开源选型穿透 license 继承链:仓库协议宽松 ≠ 权重可商用——微调权重继承底座模型协议;常见结局是真正可商用的只有数据集与方法论。
  • 重大技术路线用多条互相独立的证据链互证(学术实证 + 法规条款 + 业界工程实践),单路结论不直接进报告。
  • 量化结论按六字段落纸:区间 / 典型值 / 主驱动 / 来源 / 置信度 / caveat;联网检索到的金额只取定性,查不到的显式留白附核验路径。
04工序 · presales-tactics

售前报价与谈判:结构先于数字

与成本测算(05)的分界:那边管数字怎么算得可辩护,这边管算好的数字怎么报、怎么谈、怎么守。七条硬纪律的核心四条:

纪律一先摸底再报价

报价锚 = 规模 × 频次 × 人数,不是技术难度——同一需求在不同体量下可差 10–50 倍。按大团队剧本套小客户,一句「你说的我都没有」就全盘失信。

纪律二三档锚定自洽

保底档必须「做残」——不包主推档的真痛点功能,否则客户踩上台阶就不走;高档离客户体量太远反而摧毁整个定价可信度,只在被问时报并明说「你这体量用不上」。

纪律三砍范围不砍量级

确认体量后报一个干净数字;被砍价时给砍掉部分功能的中间档,保住价格量级——一步降十倍等于自爆虚高,定价信誉当场崩。

纪律四红线先成文再上桌

降价预案只砍功能范围与后置阶段;安全合规基线永不进砍价空间——可谈的是薪酬档位与由谁承担,法定角色与动作本身不可删。

另外三条:报价三层口径严格分离(市场价值锚 / 阶段建议价+内部底线 / 对客只报量级),小额第 0 期降成交门槛;主动降预期是信任杠杆(明说做不到的场景、先把免费竞品摆上桌再做区隔,真痛点解决不了时如实劝退);数据受托落合同(控制者归属/权限交还/不留存,话术层承诺等于裸奔)。

05工序 · bid-costing

可辩护的成本测算

成本表的每一行都会被客户逐格挑战。回答挑战的不是话术,是结构:每行数字标注四级依据(确定算式 > 官方 list 价 > 人力薪酬行情 > 估算带)与置信度;被质疑的头条数用「溯源原始出处 + 自底向上逐件重构」双管齐下互证——两者同量级则常量不改、证据落档,补证据不等于改数。

双重打折

测算云端算力成本时,把厂商「首购 5 折·限购一台」的活动价当成列表价,又乘了一次折扣系数——整套多年成本模型建立在一个根本拿不到的单价上。用户贴出购买页截图:「这已经是 5 折了。」

修正:以截图为硬锚点发起多源对抗式核价,重建「官网列表价 → 首购折(限一台,盖不住生产规模)→ 包年承诺折 → 按量时价」四级价格阶梯,由用户拍板基线后全模型重算,待实测参数列成上线前压测清单。固化的纪律:折扣只能乘在列表价上、只乘一次;二手折扣标签必须算术反算互证,对不上整行撤回。

为凑目标数调高参数

用户嫌弹性算力数量太吓人要求减半,执行时擅自把单卡并发能力上调六成来凑。用户质问:「这数据经得起推敲么?我的意思是通过降低预期用户规模来实现。」查公开基准:上调后的并发意味着单路速度慢到不可用。

修正:参数回滚并注明原值已是乐观上沿;按用户本意收紧需求侧规模档;新机制实算结果比口头目标还差一档,如实报告差距而不是硬凑。固化的纪律:降数字走需求侧,不调能力参数;容量参数须有公开 benchmark 可辩护,否则回滚并标「待压测」。

5.1范围与表格口径

  • 低~高区间 = 全低位情景 ~ 全高位情景,两端同情景配对,禁混口径;跨方案对比同会计口径。
  • 免/不免的二元情形不是区间:收敛基线值,触发条件入备注。
  • 一张表禁两轴叠加(规模档 × 低高区间):主表锁单一基准,规模档单独成曲线子表。
  • 附「未计入项」清单;可选项 = 不进合计的 +Δ 行;「X 万起」改成区间 + 典型值。
  • 影响全表的关键单价由用户拍板,拍板前冻结重算——防按错价白算一轮。
06工序 · bid-scheduling

排期:机器排出,机器校验,诚实呈报

排期表是评审逐格挑刺的对象:一处依赖倒挂、一格超载,整份材料可信度就崩。四条硬规则:

  1. 排期禁手摆
    工作包起止由带依赖拓扑约束的资源平衡器算出(附带 level.cjs,PERT 三点估算 + 每人每周人天上限)。手摆均摊曾排出单人单周 11+ 人天(上限 5)的不可行表。
  2. 产出后必做倒挂校验
    每包开始周 ≥ 全部前置完成周。曾有一版自动输出的依赖倒挂被 3 个独立评审同时标出——校验要靠算法内置,评审只是兜底。
  3. 不可行时呈分叉,不静默变通
    工期约束经平衡器证明不可行时,给「压缩硬塞 vs 放宽窗口」两选项及各自代价让用户裁决;禁止静默延长工期、静默降范围。
  4. 数据不撒谎
    目标利用率倒推只走诚实口径重定义(分母改为「有效可投产能」并显式拆解余量),不虚改底层工时。利用率对比先辨 work-bound 还是 gating-bound(合规/审批日历撑大分母)。

AI 提效写明账(原始估算 × 系数 → 净值),系数只给代码生成类工作包——ML 训练、监管审批类不吃红利,悲观值反而要拉宽。结构性溢出(腾挪 owner 只是跷跷板)按弹性范围条款把非基线可选项后置运营期,不硬塞也不留表内缺口。

07工序 · deai-writing

去 AI 味:信息零损失,只改外壳

投标语境的去 AI 味与通用检测工具关注点不同:误删领域术语、压平排名标记,比留下几个 buzzword 更致命。全程红线是信息零损失。

  1. 双层测量,拿净口径再动手
    先跑 18 类正则扫描(附带 aiflavor-scan.cjs)取毛统计;毛命中六成以上是正当用法(引用括注/表格勾选符/术语加粗),叠加语义判读扣误报、补录正则抓不到的结构性 tell(段末升华/对仗标题/拟物化)。实测净密度仅约毛数的三分之一——别追毛密度指标。
  2. 先砍性价比最高的两刀
    删段末口号式升华句;清拟物化/道德化词汇。近零信息损失,去掉最强指纹。
  3. 改写守三条红线
    排名/强弱标记、领域术语、数字与出处一律不动;「必须→建议」级的义务强度变化单独申报,不得混入「语义等价」的自称;防御性定语全文只留一处——过度自证也是 AI 味。
  4. 对抗 diff 复核(勿信改写者自评)
    由不带改写上下文的独立实例逐段专查四类:排名标记被压平、术语被软化、生造缩略词、新引入的比喻。实测改写 agent 自称「语义等价」,复核抓出四类各一处。
  5. 客户向验收 grep 清零
    禁「甲方/乙方」合同式称谓(要主语用「我们」);术语首现给定义+可核实引用;标题/封面脱离括注必须自足。
08工序 · diagram-pdf-pipeline

图表与中文 PDF:一条强耦合的管线

配图与中文 PDF 导出按「出图 → 导出 → 验收」三段执行;CJK 字体问题贯穿两域,SVG 滤镜会崩打印,图编号被 PDF 索引硬引用——所以是一条管线,不是两件事。

中文图集全空

交付物架构图用 mermaid 生成,HTML 预览一切正常,自检全部通过。用户「随便看一下」PDF:所有图的中文标签整体消失——浏览器打印 PDF 时 mermaid 不内嵌中文字体,框和箭头都在,唯独文字全空,自检极易漏过。

修正:全部图改为代码 spec 驱动的可栅格化管线(自动布局根治连线交叉、2x PNG 内嵌打印),「交付物配图绝不用 mermaid」写入长期记忆,终稿逐页人工目检从此成为铁律。

静默失败的陈旧 PDF

导出链路中加书签一步对内嵌 SVG 滤镜静默报错,只吐出一个裸括号——旧版 PDF 一直冒充新版通过了多轮「重新生成」,直到页面上出现源文件里早已删除的旧图注才暴露。

修正:逐字核对导出日志的成功行 + 重生成后内容抽验(新口径字符串出现/旧字符串消失);嵌图从带滤镜 SVG 全部换成高清 PNG(Chrome printToPDF 对 filter 直接报 Printing failed)。

8.1速查规则

  • 字体栈必须 CJK-first:Latin-first 栈的症状是「汉字正常、仅全角标点坏」(标点在部分西文字体里有字形,直接取错)。headless 环境先跑字体探测页实测可用 CJK 字体。
  • 每张图 = 一份代码 spec,同源出三件套(展示 SVG / 内嵌 2x PNG / 可编辑源);GUI 手改用语义级 diff 反向移植回 spec。
  • 编号纪律:插新图用纯新增编号(如「图 0」),绝不 renumber;近似总览图以超集为准收口单一事实源。
  • 长文档 PDF 默认带书签(Chrome 打印天然无 outline,附带 add-outline.cjs 注入)。
  • 风格层与布局层分离:连线交叉是布局问题,换自动布局引擎即可,不必丢掉已验证的 CJK/矢量能力。
09工序 · prototype-handoff

原型交接包:先吃透接收工具,再定形态

交接包不是给人看的文档,是喂给另一个工具的输入——做错形态等于白做。第 0 步永远是研究接收方:它吃什么输入?概念分层(可复用品牌层 vs 一次性原型生成层)?自身设计能力强弱?

形态接收工具特征包的主体
A:prompt + 知识库吃文件上传(含图),稠密结构化 prompt 远胜模糊描述,自带设计 craftmaster prompt + 全量真实文案 + 带借鉴注记的参考图板
B:令牌 + 组件读 tokens.css / 组件库代码,面向可复用设计系统设计令牌 + 组件规范 + 示例组件代码

形态 A 的价值序:内容完整的真实文案 > 稠密 master prompt > 带注记视觉参考 > 逐像素规范——设计令牌当护栏,不当圣经。实战翻车:对着吃令牌+组件的工具喂 prompt 形态的包,只得到很薄的仅令牌系统。

  • 合规文案逐字锁定(医疗/心理/金融/政务等强合规行业):标准化问卷题干、风险提示、免责声明、知情同意等锁定语句必须逐字提供并标注「禁止改写」——AI 设计工具会自行编造貌似合理的界面文案,生成后逐字核对。
  • 宿主视觉实测为准,不信官方 VI:真实录屏抽帧 + 像素取样,才能得出「主题色是点睛不是底色」这类结论——官方规范色与 App 实际渲染常不一致;双层令牌(--host-* / 自有品牌)隔离两套视觉。
  • P0/P1/P2 分批放行:几十屏一次生成会爆上下文且跨屏一致性崩;核心流程审过再放行下一批。
  • 交付前按用户真实打开方式跑最劣环境:双击 file:// 打开、现场断网,亲自验证。
10出厂门 · adversarial-review

对抗审校:拦下自己引入的矛盾

改写、级联、多 agent 综合——这些工作的真风险不是错字,是自己引入的隐性自相矛盾与 overclaim。实测一轮定位改写引入 6 处缺陷,全部被 4 个独立透镜抓出。审校门按交付物类型装配透镜:

对象透镜
文档类内部一致性 · 跨文档 claim 溯源 · 脱敏五类 grep · overclaim(绝对化否定句/孤儿引用)· 时效 · 口径分层
财务表算术配平独立重算(每总额=分项Σ、每差额=两方相减、中点=均值、章节对账)· 残留 grep 计数清零
视觉类逐页渲染目检(中文完整/边线路由/图不跨页)——代码跑通、子代理自检通过都不等于排版正确
全票否决的完美论文

一篇与核心技术路线结论完美贴合的论文,承载了报告中五条关键主张,险些写进客户交付物。三票独立对抗验证全票否决:无法核实这篇论文真实存在。

修正:整体剔除该来源的全部主张,同一结论改由同行评审实证、法规条款、业界工程实践三条互相独立的证据链支撑,报告末尾加透明度声明。固化的纪律:完美贴合结论的引用最可疑,必单独核验存在性。

10.1让检查器自己也过审

  • 检查器有效性反向验证:检查绿灯后注入一个已知错误,确认真能抓到,删除后复测干净——防空转假通过;报错的检查 ≠ 通过的检查。
  • grep | head 会掩盖退出码造成假阴性,清零判据用 grep -c 计数;命中逐条人工分类(「禁止出现 XX」的规则自述不是泄露)。
  • 机械可检问题(算术/超载/残留)送审前自修,评审只留判断类;≥3 个独立视角共指的发现基本必为真问题。
  • 审校结论与用户已明确给出的事实冲突时,按用户事实定稿,冲突转为置顶的现场核实项。
  • 多 agent 编排要点:子代理简报四件套(基线常量/输出 schema/来源优先级/反编造条款);schema 属性键只用 ASCII;批量生产先手写锚点样例锁约定。
