# /bid:status — 口径与红线速查 + 遗留待办

**定位**:改数前、回复客户前、新会话接手项目时的 30 秒自检。产出四件:锁定口径表、红线清单、遗留待办、漂移报告。**不做**项目进度全景、文件树浏览、git 状态汇总——那不是本命令的事;会前完整准备包(讲解脚本/模拟 Q&A/口径桥)走 `/bid:meeting --prep`,本命令只是它的速查子集。

## 固定执行序(六步)

1. **激活 skill** `bid-playbook`(口径分层与红线规则的方法论来源,尤其「合规与口径红线」一节和 `references/compliance-and-caliber.md`)。
2. **定位事实源**。按优先级读三处:项目 `.claude/memory/`(口径决策条目)→ `build/` 生成器数据源(数字的唯一真源)→ `docs/内部/` 底稿。判据:找到明确的「已锁定口径」记录才继续。**停止条件**:三处都没有口径记录 → STOP,报告「本项目尚无口径档案」,建议先跑 `/bid:init` 建档或 `/bid:meeting` 归档会议定案;**绝不**凭对话上下文或常识臆造口径表。
3. **出锁定口径表(对客/内部分层)**。双列结构:「对客已报(锁定)」列 vs「仅内部(勿口播勿投屏)」列。规则:
   - 已报客户的数字即锁定值,对客列只出现锁定值;内部测算区间/中位数只入内部列。示例(教学值):对客报 100,内部测算区间 90–120 → 对客列 = 100,内部列 = 90–120 并标红「勿口播」。
   - 两层对同一事实**有意**持不同口径的(如内部保留的成本依据),差异显式标注「分层例外:内部故意保留」,不当作待修矛盾。
   - 每个数字注明出处(memory 条目名 / 生成器源文件 / 交付物编号)。判据:每个值可回指出处;查不到出处的值标「⚠ 待核」,不照抄进锁定列。
4. **出红线清单**。从 memory 抽取全部「勿/绝不/不提/已删/已撤/统一改为」类决策,每条带三要素:红线内容 + 被废弃的旧说法 + 一句防回潮理由(为什么撤)。判据:高权重口径决策一条不漏;含分层例外(客户向已撤、内部故意保留的要写明两侧状态);拿不准是否红线的列入并标「待确认级别」,不擅自剔除。
5. **出遗留待办三清单**,每条注明来源(哪条 memory / 哪次会议定案):
   - **待实测/POC 压测项**:所有标注「待实测/待压测」的假设参数(利用率、并发系数、单价假设一类);
   - **待需求方确认项**:等对方答复的 P0 问题、责任划分、可转移成本项;
   - **遗留待核项**:memory 里登记过的文档内部残留不一致(图索引编号、脚注引用一类)。
6. **漂移抽查(只读)+ 交付**。从口径表挑最关键的数字(报价、工期、档位),grep 客户向交付物与生成器源比对。发现不一致 → 只输出差异清单(文件 + 位置 + 当前值 vs 锁定值),**不修改任何文件**,修复导流 `/bid:sync`(skill `single-source-sync` 的改源→重生成→级联→grep 残留七步);涉及 3 个以上文档的漂移额外建议 `/bid:review`(skill `adversarial-review`)收口。最后按「口径表 → 红线 → 三清单 → 漂移报告」固定结构单屏输出。

## 使用时机

| 场景 | 用法 |
|---|---|
| 要改某个已报数字,先确认现状 | `/bid:status` → 看口径表与出处 → 改动走 `/bid:sync` |
| 回复客户消息/邮件前防口误 | `/bid:status` → 只用「对客已报」列的值 |
| 新会话接手项目,快速载入红线 | `/bid:status` → 通读红线清单再动手 |
| 会前 30 秒速览 | `/bid:status`;要完整准备包 → `/bid:meeting --prep` |

## 安全保证

- **全程只读**:不改交付物、不 commit、不覆盖文件、也**不写 memory**——口径新决策落 memory 是 `/bid:meeting` 与 `/bid:sync` 的固定末步,本命令发现的漂移/疑点只登记在输出报告里,由后续命令落盘,避免速查命令自己制造口径变更。
- **发现漂移绝不顺手改**:哪怕是一处明显笔误,也只报告并导流 `/bid:sync`——单点手补产物会破坏生成器单一真源(改源→重生成才是正路)。
- **输出本身遵守分层纪律**:速查报告含内部列,属内部材料,提醒用户勿直接转发或投屏给客户。
