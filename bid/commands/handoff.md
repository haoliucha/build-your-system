# /bid:handoff — 按接收工具定制原型交接包

**$ARGUMENTS**

交接包不是给人看的文档,是喂给另一个工具的输入。**没有固定文件数的「标准包」**——同一项目对不同接收工具就该产出截然不同的包形态。所以本命令的第一步永远是研究接收方,不是打开模板。

## 执行流程

1. **激活 skill** `prototype-handoff`,把参数透传给它。参数解析:第 1 个参数 = 接收工具名(如「某 AI 原型工具」「某设计系统同步工具」),第 2 个参数 = 原型范围(如「全部 20 屏」「仅核心流程」)。缺接收工具名 → 从会话上下文推断;推不出 → **停下问用户**,接收工具是本命令的必要输入,不猜。

2. **研究接收工具的输入模型**(方法见 skill 的 references/receiver-input-models.md)。必须能明确回答三问才继续:
   - 它吃什么形态的输入?(文件上传/含图/稠密 prompt/tokens 与组件代码)
   - 它的概念分层?(可复用品牌层 vs 一次性原型生成层,各吃哪层输入)
   - 它自身设计 craft 强弱?(强 → 令牌当护栏不当圣经;弱 → 需要更硬的规范)

   判据:三问任一答不出 → 先查工具文档或让用户试跑一次,**不凭印象定形态**。喂错层的实战代价:对着吃令牌+组件的工具喂简报+prompt 包,只得到很薄的仅令牌系统。

3. **按工具定包形态**,二选一或并行两包:
   - **形态 A(prompt+知识库路)**:接收工具吃文件上传含图、稠密结构化 prompt 远胜模糊描述、自带设计 craft → 包主体 = master prompt + 全量真实文案 + 带借鉴注记的参考图板。价值序:**内容完整真实文案 > 稠密 master prompt > 带注记视觉参考 > 逐像素规范**。
   - **形态 B(设计系统路)**:接收工具读 tokens.css / 组件库代码、面向可复用设计系统 → 包主体 = 设计令牌 + 组件规范 + 示例组件代码。
   - 接收方同时有品牌层与原型生成层需求 → **分别建两包**,并在报告中说明两包各服务哪层、如何配合。

   判据:向用户一句话说明形态选择理由;理由说不清 → 回第 2 步补研究。

4. **组装必含件**(无论哪种形态,四件缺一不可):
   - **逐字合规锁定文案**:强合规行业(医疗/心理/金融/政务)的锁定语句——风险揭示/免责声明、知情同意、标准化量表/问卷题干、危机干预话术等,逐字放入包内并标注「逐字使用,禁止改写」。来源必须是项目内已裁决的定稿文案;**缺 → 停下向用户索要,绝不代拟**——AI 设计工具会自行编造貌似合理的界面文案,这是不可替代的人工输入。清单与核对法见 skill 的 references/compliance-copy-lockdown.md。
   - **全量真实 copy**:所有界面文案给真实定稿,不留 lorem/占位。包内若出现对客口径数字(报价/规模/指标),必须取自项目 build 数据源,禁止手编(单一数据源纪律,详见 skill `single-source-sync`)。
   - **实测取样的视觉参考**:宿主/参考 App 的色值与布局必须来自真实录屏抽帧 + 像素取样,**禁止按官方 VI 猜**(官方规范色与 App 实际渲染常不一致,踩过整体跑偏的坑)。没有录屏 → 向用户索要,**随附拍摄清单**列全所需模块让对方一次录全;抽帧用 skill 内 scripts/extract-frames.sh(固定节奏 + 编号 contact sheet,禁用运动检测去重)。
   - **宿主视觉双层令牌**:嵌入宿主平台的原型,tokens 用 `--host-*` 宿主 chrome 层与自有品牌层隔离,防止两套视觉互相污染。

5. **写 P0/P1/P2 分批放行说明进包 README**:P0 核心流程先生成、审过再放行 P1/P2;明确每批屏数与验收点。判据:多屏原型(约 10 屏以上)必须分批——一次生成全部会爆接收工具上下文且跨屏一致性崩;README 里没有分批计划 = 包不完整。

6. **交付前审校**:激活 skill `adversarial-review` 对包做一轮审校,重点三查:
   - 合规锁定文案与定稿逐字 diff(零改动才通过);
   - 分层泄露扫描:内部口径(内部成本、内部代号、竞品对比)是否混入——交接包会离开本机进入第三方工具,**发现即 STOP 报告**,这是泄露事故不是小问题;
   - 包内数字与 build 数据源一致性抽查。

7. **最劣环境核验**(包内含 HTML/演示物才做):按用户真实打开方式亲自跑一遍——双击本地文件(file:// 下 SVG 可能不加载 → 嵌图换 PNG)、断网(CDN 依赖备本地兜底)。判据:亲自打开并截图确认,不接受「应该能开」。

8. **落盘与交接报告**(只预览,不执行破坏性动作):
   - 包写入项目 `design/` 目录(目录约定见 skill `bid-playbook`);**目标文件已存在 → 展示 diff 预览后停止等确认,绝不直接覆盖**。
   - **绝不自动 git commit**,只给出建议 commit message。
   - 报告内容:包路径与文件清单 / 形态选择理由 / 分批放行计划 / 「本次没拍到、待补拍」模块清单 / **建议落 memory 的结论清单**(本次工具输入模型判定、实测色值与视觉纪律结论等口径级发现)——只列出,不自动写;落盘走 `/bid:meeting` 或 `/bid:sync` 的固定末步。

## 硬护栏

- 合规锁定文案缺定稿 → 停止组包等输入,绝不让接收工具或本命令代拟。
- 宿主视觉无实测素材 → 只出拍摄清单等素材,不出「按官方 VI 先凑合」的包。
- 内部口径混入包内 → 立即 STOP 报告。
- 覆盖文件 / git commit / 迁移旧包 → 一律只出预览与建议,执行权留给用户。

## 常用用法

| 场景 | 命令 |
|---|---|
| 给 AI 原型工具备包(形态 A) | `/bid:handoff 某AI原型工具 全部20屏` |
| 给设计系统工具备包(形态 B) | `/bid:handoff 某设计系统同步工具 令牌与组件` |
| 只做核心流程首批 | `/bid:handoff 某AI原型工具 仅P0核心流程` |
| 从会话上下文推断接收方 | `/bid:handoff` |
