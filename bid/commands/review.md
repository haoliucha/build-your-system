# /bid:review — 交付物多透镜收口审校

**$ARGUMENTS**

## 执行流程

1. **激活 skill 并确定审校对象**
   - 激活 `adversarial-review`(主 skill,承载全部透镜定义与裁决规则);按对象类型按需加载 `deai-writing`(去AI味透镜)、`single-source-sync`(残留判读与生成器修复链)、`diagram-pdf-pipeline`(视觉目检标准)、`bid-playbook`(岔路口请示原则)。
   - 参数给了路径 → 逐个确认存在;无参数 → 从会话上下文推断最近改动的交付物集合;推不出才问用户。
   - 给每个对象标注两个属性:**类型**(文档 / 财务表 / 视觉)与**受众层**(客户向 / 内部)。受众层决定脱敏透镜是否启用及严格度。
   - *停止条件:对象清单为空或路径全部不存在 → STOP 报告,不凭空审校。*

2. **送审前自修(机械可检问题不进评审)**
   - 评审是校验,不是替执行者做平衡:超载/倒挂/舍入溢出/算式不平/可 grep 的残留旧值等**确定性可检问题,先自行修复**,透镜只留给判断类问题。
   - *判据:确定性检查全部清零后,才允许扇出透镜。*

3. **按类型装配透镜,并行扇出独立审校**
   每个透镜是一个独立视角(并行 subagent 或独立 pass),互不通气,各自输出发现清单:
   - **文档类**:①内部一致性(同文档前后矛盾)②脱敏五类 grep(他方客户名与锁定价格 / 「规避」式表述 / 内部批注与 meta 指令 / 指向内部文件的引用 / 折扣等内部策略)③去AI味(按 `deai-writing`,信息零损失只改外壳)④准确性(易过期硬事实——型号/版本/规格/价格——联网核当前最新)⑤overclaim(「不依赖任何X/绝不Y」式绝对化否定句会锁死同文档别处留开的备选,改陈述具体正向事实;删前提后扫孤儿引用防反向泄露)⑥跨文档 claim 溯源(本文档的肯定断言不得与另一份交付物的「未公开/需核实」caveat 打架)。
   - **财务表**:算术配平独立重算——每个总额=分项Σ、每个差额=两方相减、每个中点=两端均值、章节间对账;残留旧值检查用 `grep -c` 计数为 0 作判据(**禁止 grep 管道接 head**——退出码被掩盖会造成「已清零」假阴性)。
   - **视觉类**:逐页/逐张亲自渲染目检(中文完整 / 边线路由 / 图不跨页 / 无重复标题);**代码跑通、子代理自检通过都不等于排版正确**,以目检通过为完成标准。
   - 竞品/第三方平台能力断言逐项验真:查无公开披露 → 写「官方未公开,不评判」或删除,凭体验语气的推断不得写成事实(客户一通电话即可证伪)。

4. **检查器有效性反向验证**
   - 对关键的自动检查(脱敏 grep/残留扫描),注入一个已知错误确认检查器真的在扫目标文件且能抓到,删除后复测干净——防空转假通过。
   - 检查命令因参数展开/路径问题**报错 ≠ 通过**:用显式文件列表重跑,拿到真实零命中才算过。
   - *停止条件:检查器抓不到注入的已知错误 → 先修检查器,不采信其结论。*

5. **汇总裁决(合并各透镜发现,逐条定性)**
   - **≥3 个独立视角共指**的高优先发现,基本必为真问题 → 列为必修。
   - grep 命中逐条人工分类:数字子串 / 「禁止出现XX」的规则自述 = 合法误报,报告要说明命中性质而非只报有无。
   - **完美贴合结论的引用最可疑**:单独核验其存在性,无法核实的来源即使承载多条主张也整体剔除 + 交付物加透明度声明,结论改由多条独立证据链支撑。
   - **审校结论与用户已明确给出的事实冲突时,按用户事实定稿**,冲突转成置顶的现场核实项,不盲从审校意见改稿。
   - *停止条件:发现涉及已锁定的对外口径数字(实质改变交付物形态/数字的岔路口)→ 按 `bid-playbook` 一次性收敛带推荐项请示,不自决改数。*

6. **修复与复验**
   - 逐条修复判断类问题,修完**重跑对应透镜复验清零**(定位/口径类改写自身就是缺陷高发源——修复引入的新矛盾同样要被抓)。
   - 任何义务强度/口径层面的语义变更**单独申报由用户拍板**,不得混入修辞修复并自称语义等价。
   - 修复若触及生成器产物(xlsx/图/PDF)→ 走 `single-source-sync` 的改源→重生成→残留 grep 链,**绝不直接手改产物文件**。

7. **交付报告 + 预览(不自动执行 destructive 动作)**
   - 输出发现对照表:问题 / 命中透镜 / 裁决(必修/建议/合法误报/留用户定夺)/ 处置结果。
   - **commit、覆盖重生成产物等 destructive 动作一律只列预览清单**(显式文件路径 + 排除集),等用户确认后才执行。用户可选「只记录不改」→ 挂待办,交付物一字不动。
   - 若审校产出口径级决策(新锁定数字/新红线/新例外),提示用户走 `/bid:sync` 或会议纪要将其落 memory 防回潮——memory 维护是 meeting/sync 的固定末步,本命令只提示、不代写。

## 常用用法

| 场景 | 命令 |
|---|---|
| 收口审校指定交付物 | `/bid:review docs/01-方案书.md docs/05-成本测算.xlsx` |
| 审校整个客户向目录 | `/bid:review docs/客户向/` |
| 刚做完一轮口径改写 | `/bid:review`(从会话上下文取最近改动集) |

## 安全保证

- 只读审校 + 本地修复,**绝不自动 commit / 不自动覆盖用户手改过的产物**(手改检测见 `single-source-sync`)。
- 外部页面抓不到(登录墙/被拦)→ 用已验证的既有材料或 STOP 报告,绝不围绕抓不到的内容编造。
- 查不到的事实显式列「需进一步确认」清单并给核验路径(官方渠道/现场摸底),禁止推测填空。
