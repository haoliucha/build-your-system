# 接收工具输入模型调研法

交接包的形态由**接收工具的输入模型**决定,不由交付习惯决定。跳过这一步的典型结局:包做成了工具读不了或喂错层的形态,全部返工。

## 调研清单(动手前逐项回答)

1. **输入通道**:支持文件上传吗?上传含图吗?能贴多长的 prompt?能读代码目录/仓库吗?有没有持久化的知识库/项目空间(上传一次、多轮生成共用)?
2. **概念分层**:工具把「可复用的品牌/设计系统层」和「一次性的原型生成层」分开吗?各层吃什么形态的输入?——这是 A/B 形态判定的关键。
3. **上下文预算**:单次生成能吃多少材料?超了会静默截断还是报错?(决定分批粒度)
4. **自身设计能力**:工具会自主做设计 craft(排版/留白/动效)吗?会 → 逐像素规范是浪费,给方向和护栏即可;不会 → 才需要更细的规范。
5. **输出形态**:产出代码、图、还是可交互原型?(决定交付核验方式)

## 怎么调研

- 读官方文档、发布说明、最佳实践文章;搜工具方自己发的「如何喂输入」类指南。
- **小样试喂**:先用 1 屏的最小材料跑一次,观察工具吃进了什么、丢了什么、自作主张了什么,再决定包的结构与颗粒度。比通读文档更可靠。
- 注意版本时效:AI 设计工具迭代快,以当前版本实测为准。

## 形态判定

| 信号 | 走形态 A(prompt+知识库) | 走形态 B(令牌+组件) |
|---|---|---|
| 输入通道 | 文件/图片上传 + 长 prompt | 读 tokens.css、组件源码 |
| 服务对象 | 一次性原型生成 | 可复用设计系统/品牌层 |
| 包的主体 | master prompt + 全量真实文案 + 参考图板 | 设计令牌 + 组件规范 + 示例组件 |
| 示例模板 | references/package-a-template.md | 按目标工具的设计系统约定组织 |

## 两包配合(同一项目可能都要)

实战教训:同一项目里先做了 A 形态包(简报+prompt),后来又出现走「令牌+组件」同步命令的路径——直接拿 A 包对着跑,只会得到很薄的仅令牌系统。正确做法:

- **B 包喂品牌层**:一次性同步进工具的设计系统,跨屏/跨轮次复用。
- **A 包喂生成层**:每次生成原型时作为主输入。
- 令牌变更时先更 B 包再更 A 包引用,避免两包漂移。
- 向用户明确说明两包各自用途与使用顺序,不要默认对方能看出来。

## 价值序的依据(形态 A)

**真实文案 > master prompt > 参考图板 > 逐像素规范**:

1. **内容完整的真实文案**是工具无法编造的部分(尤其合规锁定文案,见 compliance-copy-lockdown.md),缺了它工具只能填 lorem ipsum 式的貌似合理文案,原型评审价值归零。
2. **稠密结构化 master prompt** 远胜模糊描述——把屏清单、信息层级、不可协商项(锁定文案、宿主令牌)、可自由发挥区都写明。
3. **带借鉴注记的参考图**每张标注「借鉴什么/忽略什么」,防止工具整图照抄错误维度。
4. **逐像素规范**对有 craft 的工具是浪费:它会重新诠释,死抠数值反而打架。设计令牌当护栏(色板/间距体系不越界)不当圣经。
