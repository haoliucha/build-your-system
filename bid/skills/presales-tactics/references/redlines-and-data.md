# 谈判红线与数据受托

红线的定义:**在任何降价压力下都不进砍价空间的项**。先成文,再上桌。

## 1. 安全合规基线永不砍(红线之首)

- 法定合规项(等级保护、行业备案、数据保护义务、法定安全角色配置等)是**基线,不是配置项**——砍掉它们的报价在验收/监管环节必然爆雷,砍下来的钱会加倍还回去。
- **可谈的维度**与**不可谈的维度**要拆开:角色与动作本身法定刚性、不可删;可弹的是承担方式——薪酬水平档位(初级 vs 资深)、由谁出人(我方派驻 vs 客户自有在编人员承担,后者我方可计 0 并注明转移条件)。降价谈到合规项时,把话头引到"谁承担",而不是"做不做"。
- 对客表述模板:"这部分是法定要求,报价里砍不掉;但如果贵方有自有人员满足资质,这块可以由贵方承担、从我方报价移除。"

## 2. 红线文档要素(上桌前写好)

谈判前成文一页红线清单,谈判者不临场发明:

1. 价格底线(来自三层口径的第②层,绝不出现在对客材料)
2. 不可砍项清单:合规基线 / 数据保护条款 / 质量门(测试与验收动作)
3. 可砍项及顺序:功能范围 → 后置阶段(降级树见 anchoring-and-tiers.md §4)
4. 每个可砍项对应的价格降幅上限(单步不破量级)
5. 触发离席的条件(如要求砍合规项且不接受转移方案)

## 3. 数据授权是先于代码的第 0 步(高权重)

**反模式 → 正确做法**:把数据授权当交付中后期的法务收尾。→ 以客户存量数据为核心资产的 AI 项目,**历史数据采集时有没有"可用于 AI/研究"的授权**是第 0 个问题,列入首次会谈 P0 提问清单。答案直接决定技术方案分叉:

- 有授权 → 数据可直接入库/入训练管线;
- 无授权/授权不明 → 管线必须走"脱敏 + 合成重建 + 人工质检"路线(业界通行做法:没有人直接拿无授权的原始真实数据训练)。

这个问题晚问一周,方案可能重写一半——所以是第 0 步,先于任何代码与架构承诺。

## 4. 数据受托关系落合同

**反模式 → 正确做法**:"数据是你们的、我只帮你配"只在话术层承诺。红队判定:作为持管理员权限的受托处理方,无书面约定 = 纯合规地雷,话术承诺等于裸奔。→ 只要受托配置或以管理员权限接触含个人信息的客户数据,合同必须含:

1. **控制者归属**:数据控制者始终是客户,我方仅为受托处理方;
2. **权限交还**:交付完成后管理员权限移交/注销的时点与方式;
3. **不留存不外传**:我方不留存副本、不用于其他用途;
4. 风险声明配套三条:个人信息处理告知义务归属 / 平台账号存续与数据取回路径 / 产出物的合规认定责任归属;
5. 所有承诺的前提限定在**客户实名开立的企业租户**内——个人账号代持场景不承诺。

## 5. 自建档位写明数据保管负债

报价中含"客户自建/私有化"档位时,方案里明写:自建意味着客户承接**长期数据保管负债**(备份、安全加固、合规审计、人员离职交接)。这既是诚实降预期(见 trust-levers.md §1),也防止客户日后把保管事故归因于建设方。
