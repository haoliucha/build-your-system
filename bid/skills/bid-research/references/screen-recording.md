# 录屏拆解详解

## 为什么固定节奏抽帧,不用帧差/场景检测

`ffmpeg -vf "select='gt(scene,T)'"` 这类场景切换检测的前提假设是「界面切换 = 大像素变化」。移动应用录屏会击穿这个假设:

- **滚动列表**:滚一屏产生的像素变化量可以超过一次真正的页面跳转 → 阈值调低时爆出海量冗余帧。
- **转场动画/轮播/粒子背景**:持续的小幅变化把「真切换」淹没在噪声里 → 阈值调高时漏掉关键界面。
- 阈值没有普适取值,每段录屏都要重调,调参时间超过收益。

固定节奏(默认 1fps;操作很慢的录屏可 0.5fps,操作密集可 2fps)保证时间轴均匀覆盖,冗余帧交给阅读 contact sheet 时快速跳过——agent 多读几张缩略图的成本,远低于漏帧漏掉关键界面的代价。

## 标准流程

```bash
bash "${CLAUDE_PLUGIN_ROOT}/skills/bid-research/scripts/extract-frames.sh" <录屏.mp4> <输出目录> [fps] [cols]
```

产物:`frames/f0001.png ...`(全分辨率原帧)+ `sheets/sheet_00.png ...`(带帧号标签的 contact sheet)。

1. **自检(必做,不可跳)**:Read 第一张 sheet,确认两件事——帧号标签渲染出来了、缩略图分辨率足以辨认界面。任何一项不满足,先修再批量读。
2. 批量 Read 全部 sheets,建立**分镜表**:帧号区间 → 场景名(如 `f0001–f0012 启动与权限弹窗`)。
3. 关键场景回读 `frames/fNNNN.png` 全分辨率原帧,锁定精确文案(按钮字样、弹窗全文、价格数字)。
4. 写拆解,每条结论带帧号引用,格式建议 `f0083 ≈ 82s`(fps=F 时 fNNNN 的 N 对应约 (N-1)/F 秒)。

## macOS 字体坑(踩过)

`magick montage -label` 在 macOS 上不显式指定 `-font` 完整字体文件路径时,标签可能**静默空白**——字体 fallback 失败不报错,contact sheet 照常生成,只是没有帧号,时间定位能力全失。脚本已默认 `/System/Library/Fonts/Helvetica.ttc` 并逐级 fallback;其他平台用 `FONT=` 环境变量传入。这就是「先自检一张再批量读」的由来:批量读完才发现没帧号,全部白读。

## 取证三档边界

| 档 | 能写什么 | 引用方式 |
|---|---|---|
| 录屏可实证 | 界面、流程、文案、有无某功能/提示 | 帧号 + 时间 |
| 公开资料可证 | 备案、资质、运营主体、公开报道 | 链接 + 截图 |
| 推断 | 架构、模型形态、后端实现 | **不落纸**;确要写则显式标「推断」,且绝不作为攻击点 |

反面教训:某 To-B 投标项目中,拆解报告里所有基于界面表现推断的「对方模型形态」断言,终稿前被全部删除——从一开始就不该写。

## 合规关键场景追踪

竞品拆解不止列功能。主动在录屏中寻找(或请用户按拍摄清单构造)**高风险/合规兜底时刻**,记录产品的真实响应路径:有无干预、有无提示、有无转介,还是直接推进商业转化。

- 实证到的短板(如:高风险输入无任何兜底提示、直接推进转化流程)→ 这是方案里最有力的**正面差异化打点**,因为它帧号可查、无法抵赖。
- 没实证到的 → 不写。这一类打点的全部价值就在「可实证」三个字上。
