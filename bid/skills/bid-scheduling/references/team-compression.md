# 压缩团队与能力缺口

## 压缩团队的完整评估法(高权重教训)

**反模式**:用户问"人力成本高、利用率又偏低,评估压缩方案"时,只回"砍掉某角色可省 X"——这不是评估,是拍脑袋。

**正确做法**:一次给全六件,先评估获确认,再落地重算排期。

1. **筛角色**:按「个人利用率 × 是否在关键路径峰值上 × 是否承载核心卖点」三维筛,砍利用率最低且不在关键路径峰值、不承载卖点的角色(常见是可被相邻角色吸收的辅助岗)。
2. **活的去向(shift-left)**:被砍角色的工作按性质左移分摊给有余量的相邻角色,逐项列「原 owner → 新 owner」对照表。
3. **验证峰值并行期**:重跑平衡器确认峰值周的专才并行不受损——峰值需要多专业同时在场是压缩的硬下限。
4. **量化前后对比**:新利用率、节省人月/金额、储备率变化,全部带数字。
5. **再砍下限论证**:说明为什么不能再砍(峰值并行/单点风险),堵住"再砍一个"的追问。
6. **预答评审反驳**:最可能的一问是「法定合规角色是不是被砍掉了」。标准答案:法定专职岗位(如安全管理员)属**运营期**的另一笔账,与**建设期**减员不冲突;建设窗口内相应职责由谁临时覆盖,写明。

注意:压缩后常触发结构性溢出(见 repair-playbook.md),按弹性条款后置可选项,不在角色间跷跷板。

## 卖点 UI 缺口:拆工程与美术

当卖点级 UI/交互超出现有团队能力时,不要笼统提"招一个专业前端"。把「专业前端」拆成两类缺口分别决策:

- **前端工程**(组件/状态/联调):留在自有团队,AI 辅助编码可覆盖;
- **视觉美术**(视觉规范/插画/动效资产):外包为**可复用资产包**,一次性买断。

排期与成本上的落法:

- 外包资产到位周作为相关前端包的**排期前置依赖**(任务表里用 `earliestStart` 或建一个里程碑让前端包 deps 引用);
- 外包费用列为**独立一次性成本行**(与 bid-costing 对接),不摊进人力。
