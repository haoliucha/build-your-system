# 客户向验收规范

## 称谓:禁「甲方/乙方」(高权重教训)

**反模式**:某 To-B 投标项目全套交付物数十处「甲方」,用户 review 判为「完全没必要提,太见外太生硬」。

**正确做法**:
- 要主语就用「我们」;
- 纯所有格(「甲方的系统」)直接删所有格;
- 归属声明改「完全自有」「自主可控」类表述;
- 先向作者确认口径(是否全文不区分甲乙丙方),确认后一次推平全部交付物。

**验收命令**(grep 清零才算过):

```bash
! grep -rn -E '甲方|乙方|丙方' docs/ && echo PASS
```

## 术语/量表/法规缩写:首现定义 + 核实引用

- 专业量表、法规缩写、行业术语(TCO 类成本缩写、测评量表名、标准编号)**首现处**给一句定义 + reference(学术原始文献 / 国标编号 / 官方文档);
- 引用**先联网核实再写入**;不确定的引用宁可不加——一条错引用比没引用伤害大;
- 可在附录集中设「参考与依据」块。

**反模式**:文档默认读者懂行话,评审连问「这个缩写是什么意思?提供 reference」——每一问都是可信度扣分。

## 标题/目录/封面:脱括注须自足

章节标题、目录、封面会**脱离正文括注单独显示**。正文里靠括注澄清的词,在标题里会被读成更强的主张——例如孤立的「自建」在招投标语境易被读成「从零自研」,构成轻度 overclaim。

**正确做法**:标题与目录让关键限定词露出(如「开源组件私有化自建」);正文带括注处保留原词。改标题后重新生成目录/封面,三处一起验。

## 按转发链条终端读者写

对外材料的直接收件人常会把它转给真正的评审人/决策人。按**转发链条末端读者**优化:

- 末尾附与目标项目的**逐项契合映射**(需求点 ↔ 我方能力,这是全文最值钱的段落);
- 主动备好精简版选项(终端读者时间更少);
- 不依赖"收件人会替你解释"——文档必须自足。

## 环境备注(适用条件)

- 扫描脚本零第三方依赖,Node.js ≥ 16,macOS/Linux 均可直接运行。
- 若交付物同时存在派生形态(PDF/幻灯片由 md/html 生成),改完源文件后**重新生成派生物**,只改源不重生成 = 验收假通过。
