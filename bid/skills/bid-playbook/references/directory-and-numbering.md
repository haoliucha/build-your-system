# 目录、编号与多项目治理细则

本文全部案例来自某 To-B 投标项目,已抽象化;示例数字均为教学值。

## 1. 标准目录树与各目录纪律

```
<project>/
├─ meeting/
│  ├─ 2026-01-05-首次需求沟通.md
│  ├─ 2026-01-12-方案评审.md          # 按日期编年,永不按主题/项目拆子目录
│  └─ ...
├─ docs/                              # 客户向,交付时整目录可直接打包外发
│  ├─ 01-方案书.md / .pdf
│  ├─ 02-需求分析.md
│  ├─ 02B-需求细化.md                 # 后补文档:母文档号+字母后缀插位(见 §2)
│  ├─ 03-架构设计.md
│  ├─ 04-实施排期.xlsx
│  ├─ 05-成本测算.xlsx
│  └─ 内部/                           # 物理分离,绝不外发
│     ├─ 05B-成本底稿.md
│     └─ 06-报价依据.md
├─ build/                             # 生成器:每个含数字的交付物对应一个脚本
│  ├─ gen-budget.cjs                  # → docs/05-成本测算.xlsx
│  ├─ gen-plan.cjs                    # → docs/04-实施排期.xlsx
│  └─ gen-report.cjs                  # → 01 的精排 HTML/PDF
└─ .claude/memory/                    # 口径决策即时落盘
```

纪律:

- **meeting/ 按日期共享编年**。判断依据:客户侧常是同一批人跨多个项目开会,按项目切分 meeting 目录会把一场会撕成两半。多项目时在每篇纪要头部打「涉及项目: A / B」标签,而不是拆目录。
- **docs/ 与 docs/内部/ 的物理分离是故意设计**,目的有三:①外发/投屏时按目录整体操作,杜绝挑错文件;②内部文件可保留完整行情区间、折扣策略、谈判底线;③两层对同一事实可有意持不同口径(见 `compliance-and-caliber.md` §4),分离让这种不一致成为受控设计而非 bug。
- **build/ 是数字的唯一真源**。交付物 xlsx/PDF/图一律由脚本生成,手改产物必须回捕落源(细节见 `single-source-sync` skill)。
- 重命名或重组目录前,**先审计所有生成脚本的硬编码路径**,确认改动是机械且局部的(如仅头部路径常量)再动手,重组后跑一遍全部生成器验证管线不破。

## 2. 交付物编号:按依赖位插入,不按产出先后

规则:

- 序号反映**依赖与阅读顺序**——读者应能按 01→02→03… 顺序读完整套材料,每份文档只依赖前面的。
- 后补文档用**字母后缀插位**:`NB` 体例,紧跟其所展开的母文档。需求细化是需求分析(02)的展开,就编 02B;不因为它是第 6 个产出的就顺排 06。
- 客户向与内部**共享同一总体序号体系**(内部文档也占号),避免两套序号相互指认混乱。
- 既有编号**绝不 renumber**:交付物之间有嵌入、索引、跨文档硬引用,重编号的级联破坏远大于「序号不连续」的美观损失。全新的前置总览图/文档可用 0 号纯新增。

失败案例(反模式→正确做法):

> 反模式:为后补的需求细化文档顺手取下一个空闲序号,排到了成本/报价文档之后。用户纠正:「按正常流程,需求细化应该随需求分析一起出。」
> 正确:改用母文档号+字母后缀(02B)插入需求分析旁,并在计划里补记排序理由。

## 3. 命名与备选方案标签

- 不在交付物与汇报中使用**未定义的口头简称**——人工验收时用户的第一个问题往往就是「这个词指什么?」。命名分层统一:内部文档一律用真实项目名,客户向一律用脱敏名;改名后全库(含生成器封面文案、文件名)同步,并 grep 双向复检不泄露。
- 方向一旦锁定,正式交付物中**删除未选方案的编号标签**(「方案A/方案B」)与对照条款,改为自解释的事实陈述;备选方案定义只留在会议纪要作决策留痕。跨文档引用的备注改写为自解释措辞,不逼读者回翻内部材料。

## 4. 多主体 / 多项目治理

同一供应商服务多个独立运营主体时:

- 方案、报价、交付物目录**按项目对称拆分**,每份文档显式声明项目独立性(独立主体、独立报价、互不构成承诺)。
- meeting/ 保持共享编年+涉及项目标签(见 §1)。
- 跨项目共用的资产(公司资质、通用组件介绍)单独进共享目录,不复制进各项目。
- **新业务线索先隔离**:与当前项目无关的新线索,产出文件先留临时目录(scratchpad 或独立 tmp 目录),确认成单后再建正式子项目目录与 memory。理由:塞进现有项目目录会造成跨项目上下文与记忆污染。

## 5. 项目 memory 维护

- 每次口径/定位/呈现决策落地后**立即写入项目 memory**:索引行(一句话)+详情条目。
- 详情条目必须记录:**被废弃的旧口径**(防未来会话把已撤销的说辞重新引入)、**保留的分层例外**(如「客户向已撤某前提、内部故意保留」)、锁定数字与其生效日期。
- memory 与交付物数字的一致性,在每次 `bid-sync`（`/bid:sync` / `$bid:bid-sync`）收尾时核对一遍。
