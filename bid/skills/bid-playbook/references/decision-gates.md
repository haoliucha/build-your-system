# 岔路口管理、执行节奏与提交纪律

本文全部案例来自某 To-B 投标项目,已抽象化;示例数字均为教学值。

## 1. 真分叉判据:什么问,什么不问

| 类型 | 判据 | 动作 |
|---|---|---|
| 真分叉 | 会实质改变交付物形态、数字或范围,且无法从既有材料/调研推断 | 收敛成带推荐项的选择题,一次性请示 |
| 假分叉 | 机械可推导、有既定规范、或调研能给出答案 | 自决推进,阶段完成短报告后直接继续 |

典型真分叉:部署形态、数据授权路径、测算规模档位、交付形式(讨论稿 vs 精确报价)、编号方案、表格形态、两图取舍、素材由谁提供。

操作要点:

- **动笔前并行探明落点再问**。写方案或落地批量 review 意见前,先探明每条意见在全部交付物的确切落点与级联面,把几十个落点收敛成 2-4 个真岔路口一次问清,并明说「其余我已能推断,不打扰」。
- **每个选项带理据与推荐项**。设计类分叉先做研究再问:某次品牌主色分叉,先调研出「60/30/10 点睛式调和」方案,再把「宿主平台官方色主导多少 vs 自有调性主导多少」交用户裁决——而不是替用户默认一条。
- 改造存在多种呈现方案或「是否补调研」分叉时,动手前列选项请用户敲定,明说是为避免返工——这属于允许停下的「需要用户输入」场景。
- 执行参数分歧(尺寸口径/数量目标/保留范围)开工前一次锁定,不边做边逐个回头问。

## 2. 大交付物流程:spec → 确认 → plan → 执行

1. 先写设计文档(spec),定四类决策:目录结构、内容范围、成本模型、人力口径。
2. 用户确认 spec 后,拆成**带验收标准的逐任务实现计划**。
3. 按依赖顺序执行,每任务独立提交并双重 review。
4. 文档项目把 TDD 节奏适配为「产出→校验→提交」:校验=生成器重跑、grep 断言、逐页目检。

## 3. 宽级联前锁完整取值表

用户只改部分条目却会触发全链路级联(成本表→报价→汇总→叙述文档→memory)时,先列**完整取值表**请用户逐项确认,未提及的条目不擅自猜:

| 条目 | 当前值 | 用户新值 | 状态 |
|---|---|---|---|
| 开发人力 | 100 | 120 | 用户已改 |
| 运维 | 20 | — | 未提及,维持 |
| 部署 | 30 | — | 未提及,维持 |
| 合计 | 150 | 170(推导) | 待确认 |

锁定后才写计划执行。配套技巧:客户向叙述文档把排期写成**阶段级**、成本写成**方向性表述**(不硬编码周数与金额),数字口径反复调整时它们零改动、无需重排版重出 PDF,大幅收窄级联爆炸半径。

## 4. 指令顺序会固化矛盾时:提替代顺序

反模式→正确做法:

> 反模式:用户要求「先 commit 我手改的表格,再更新下游」,但手改版自相矛盾且与源脚本/其他文档脱节——直接照办会把不一致状态锁进版本库。
> 正确:提出并执行保一致性的替代顺序——手改意图落源→干净重生成→级联全部下游→独立校验→单个一致 commit;向用户说明为何没按字面顺序,并**保留按原样拆分的选项**(如 reset 后分两次提交)。

原则:尊重用户意图,不盲从字面顺序;替代方案必须让用户可一键退回原指令。

## 5. 意见答复与讲解纪律

- 回应逐条评审意见时,用「每条意见→处理结果+依据」对照表汇报;在级联改下游文件前留一个**显式确认口**(尤其当用户指定了先后顺序时)。
- 向用户讲解表格/模型数字时,**先用代码把每一步精确复算一遍**与表对账,再逐行拆公式与假设,点明哪些是硬数据、哪些是可替换假设;不凭记忆转述。
- 被问「X 是不是必需品」这类将军式问题时,拆成两层作答:硬约束必需层(技术/合规前提)与价值最大化选择层,并给出不押注工期的回退路径。

## 6. 效率与上下文工程

- 后台长任务(调研/生成)运行期间,利用等待时间预读将被修改的文件现状、定位全部改点,结果一回来即可一次性精改落地。
- 子任务结果在对话内被截断时,从其**落盘的结构化输出文件**定点截取缺失部分,不重跑任务;巨型 transcript 类文件禁止整读入上下文。
- 素材/截图采集以「能支撑文档论点的覆盖度」为终点,达到即停,不为流程完整性无限试错;评估边际价值再决定是否继续。
- 大范围重排+几十处散改时,**整文件重写**比大量脆弱的字符串编辑更安全。

## 7. 提交与工作区纪律

- 客户交付物改动**默认不 commit**,任务完成即完整报告,commit 动作留给用户确认;首次修复以新文件名并存原件供对比。
- 只 stage 本任务产出的明确文件清单,**禁用 add -A**;工作区内其他会话或预存的无关改动逐一点名并明确不碰。
- 大改动分组提交:生成器/交付物/memory 各一组。反模式:把旧文件的 git rm 误捆进第一组 commit,导致第二组因 stale pathspec 失败。
- 共享的 memory 文件提交前,先查其他会话是否已提交,避免重复或错误归属。
- 现场演示类网页材料若依赖外网资源(CDN 字体/图标)——适用条件:现场网络不可控时——会前检查清单写明网络方案(手机热点)与断网降级表现,并提前实测。
