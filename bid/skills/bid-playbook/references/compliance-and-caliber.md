# 数据合规前置、口径锁定与分层脱敏

本文全部案例来自某 To-B 投标项目,已抽象化;示例数字均为教学值。

## 1. 数据授权是先于代码的第 0 步(weight=high)

以客户存量数据为核心资产的 AI 项目,第 0 个问题是:**历史数据当年采集时,知情同意是否覆盖「用于 AI/研究」这类新用途?** 答案决定整条数据管线是走「脱敏+合成重建」还是可直接入库——它先于一切架构与代码决策。

反模式→正确做法:

> 反模式:先设计数据管线与训练方案,把授权问题当「之后再补的法务细节」。
> 正确:①把该问题列为首次会谈 P0 提问(用软措辞问出,如「当年的同意书里有没有科研/产品改进条款」);②调研佐证:业界通行做法是「去标识化+专家校验+LLM 合成重建+人工质检」,无人直接拿原始真实数据训练;③同一结论同时写入调研报告、谈话手册 P0 清单和项目 memory。

**未知前置条件按最坏情形设默认路径**(weight=high):客户暂时答不上的授权/合法性问题,方案默认按最坏情形走安全路径(原始数据永不直接进训练),使推进不依赖客户答复;把兜底设计包装成专业度卖点而非负担。

## 2. 未知客户事实不阻塞

售前阶段问不到的客户事实(平台、预算、拍板人、真实规模、上线窗)不当阻塞项:统一压成一张**摸底问题清单**(8-10 问)作为交付物的一部分带去现场,准备工作不停。

首次商务会谈前预设「散场必得三样」:下一步会谈时间、保密协议+脱敏样本数据、决策链信息。会谈成败以拿到清单为准,而非讲得好。

## 3. 口径锁定纪律

- **已报客户的数字即锁定口径**(weight=high)。一旦某组数字报给客户,即成硬约束:对外文件一律用它;内部行情区间/中位数只留内部文件,并标注「勿外发勿投屏」;会前红线清单显式写入「别口播内部数」。反模式:对客一页纸与内部详表各自演化,客户一对比即露馅。
- **锁定口径清扫按文档类别全量执行**(weight=high)。某措辞/数字口径锁定后,清扫范围=「全部客户向文档」这类**文档类别**,不是任务碰巧提及的几个文件。反模式:去敏任务只覆盖任务提到的两份文档,漏了同为客户向的排期表,直到改别的问题时才偶然发现。正确:发现一处泄漏,立即对同类全部 grep 排查清零,只留明示保留的良性例外,并记入 memory。
- 数字变更的完整级联链(改源→重生成→grep 残留→memory)由 `single-source-sync` skill 与 `/bid:sync` 承载,本文只定红线。

## 4. 客户向 / 内部分层脱敏

**对外演示件与对内作战手册成对产出、从第一天物理分离**:对外件不含底价与谈判策略;对内手册(定位/议程/追问清单/异议应对/报价底线/红线)绝不外流。会前准备包五件套(讲解脚本/数字速查卡/多视角模拟 Q&A/「别说」红线/口径桥)由 `/bid:meeting` 生成,本 skill 只留原则。

**客户向文档交付前的五类审计清单**(weight=high),逐类 grep 取证并清零:

1. 机密事实:他方客户名称、锁定价格、安全评级等;
2. 「规避」式表述:改为合法的事实陈述;
3. 内部批注与 meta 指令(「对客不可宣称…」这类);
4. 指向内部文件的引用(把客户引向成本底稿);
5. 报价折扣等内部策略泄露。

合法引用(如公开内容来源)作显式例外保留,例外必须点名。

**分层可以有意不一致,但必须是受控设计**:客户向撤掉的前提(如内部沿用的供应商折扣假设)可在内部文档故意保留,作为锁定报价的成本依据。要求:①显式报告这种不一致是设计而非疏漏;②是否把改动推进到内部文档由用户拍板(会牵动锁定数字);③落 memory 防未来会话把已撤销口径重新引入。

**同一数字底座分层出双版本**:成本类交付物从同一生成器分出「可调参数模型版」(细拆、活公式、防追问)与「决策者一页版」(大白话、区间+典型值);一页版必须显式点明未含项(尤其核心运营人力)与待定口径(含税/谁出钱)。

## 5. 结论治理

- **前提变化时重构结论,而非硬守或盲从**(weight=high)。业务定位或部署前提变化推翻先前技术结论时,以新前提为轴重新推导自洽论证。案例:调研结论为「A 技术为主、B 为辅」,用户新目标(单客户私有化、数据不出域)要求 B 当主角——正确做法不是硬守旧结论,也不是无理由翻转,而是从新前提出发重构叙事,使 B 名正言顺成为核心、A 降为配角,整条论证自洽。
- 对标产品先**独立核实真实定位**再做方案:客户对对标的理解常有偏差(以为是轻量工具,实为老牌企业级厂商、护城河在合规资质),纠偏结论会直接改变方案重心。取证方法见 `bid-research`。
- 某功能触发重合规义务(强制备案/审查/弹性算力)时,拆成「规避触发的轻方案」与「承担全义务的重方案」两个选项供决策者选,并建议先轻后重的进场顺序。
- 谈判红线先成文再上桌:预算被砍时只砍范围或后置阶段,**安全与合规基线绝不砍**;不承诺监管敏感领域的效果类表述。结构设计细节见 `presales-tactics`。
