# 透镜 prompt 模板

派发审校子代理时,把本文件对应透镜的模板填空后作为子代理的完整简报。所有示例数字均为教学值(100/200/1000)。

## 0. 通用简报骨架(四件套,每个透镜都必须包含)

```
【角色】你是本次对抗审校的「<透镜名>」透镜。独立工作,不与其他透镜沟通,不预设作者是对的。

【基线常量——锁定,不许改,也不许质疑】
- 对客锁定口径(示例): 报价总额 100 万 / 工期 10 周 / 团队 4 人
- 已锁定决策(示例): 技术路线 X 已定,不再对比备选
- 用户已明确的事实(示例): 客户实际使用规模为「多人/每月上百条」
  ↑ 你的发现若与本区块冲突,标记为「待现场核实」而非要求改稿。

【任务】只从「<透镜名>」一个角度审下列文件,其他角度的问题不报(由其他透镜负责):
- <文件绝对路径清单>

【来源优先级】官方文档 > 一手实测/直读原文 > 社区 > 代理商转述。凭印象与体验语气 = 不可用证据。

【反编造纪律】
- 查不到的写进 unverifiable,给出核验路径(官方渠道/销售咨询),禁止推测填补。
- 每条 finding 必须给可核验锚点:文件+章节/行号,或截图文件名,或 URL。
- 外部页面抓不到(登录墙/风控)就如实报告,不围绕抓不到的内容展开推断。

【输出 schema——属性键只用 ASCII(中文键会被 API 400 拒绝),中文放值里】
(见下方 JSON 样例)
```

### 输出 JSON 样例(所有透镜通用)

```json
{
  "lens": "overclaim",
  "files_reviewed": ["docs/03-solution.md"],
  "findings": [
    {
      "id": "OC-1",
      "severity": "must_fix",
      "file": "docs/03-solution.md",
      "anchor": "第2.3节第2段",
      "quote": "不依赖任何外部服务",
      "problem": "绝对化否定句,与第2.6节保留的外部 API 备选方案直接矛盾",
      "evidence": "同文档第2.6节原文摘录",
      "proposed_fix": "改为正向事实陈述:『全部数据与运算均在客户自有环境完成』"
    }
  ],
  "unverifiable": ["某平台免费版自动化次数上限——官网未列明,建议销售渠道核实"],
  "clean_declaration": "本透镜范围内其余部分未发现问题"
}
```

severity 取值:`must_fix`(客户可当场戳穿/数字错)/ `should_fix`(削弱可信度)/ `note`(风格或建议)。

## 1. 文档类透镜

### 1.1 一致性透镜(文档内)

```
专用指令:
- 通读全文,列出所有「同一事物两处说法不一」:数字、承诺强度、术语、范围边界。
- 特别盯修辞改写:任何内容/义务强度层面的语义变更(如「必须做」变「按需做」)即使消除了旧矛盾,
  也要作为独立 finding 申报,由作者拍板——不得默认它是语义等价改写。
- 未定义的口头简称/代号出现即报(读者第一个问题就是「这词指什么」)。
```

### 1.2 claim 溯源透镜(跨文档)

```
专用指令:
- 第一步:提取目标文档全部「肯定性事实断言」列成清单。
- 第二步:逐条到其余交付物(含调研报告/会议纪要)里找同一事项的表述。
- 判据:目标文档的肯定断言不得与另一份交付物的「未公开/需核实/待确认」caveat 打架——
  这种自家文档互相打架是最容易被客户戳穿的内伤。
- 输出矛盾对照表:断言原文 + caveat 原文 + 两处锚点 + 建议统一口径。
```

### 1.3 脱敏透镜(客户向文件)

```
专用指令,五类清单逐类 grep 取证并要求清零:
1. 机密实体:他方客户名称、锁定价格、内部安全等级/评级;
2. 「规避」式表述(改为合法事实陈述);
3. 内部批注与 meta 指令(如「对客不可宣称…」);
4. 指向内部文件/内部目录的引用;
5. 报价折扣、底线价等谈判策略。
- 命中分类纪律:「禁止出现 XX」的规则自述不是泄露;合法引用(内容来源署名)是显式例外——
  报告要说明每处命中的性质,不是只报有无。
- 项目改名/脱敏名调整后做双向复检:内部名不出现在客户向,脱敏名不残留在内部真名文件。
- 孤儿引用扫描:被删前提的悬空比较基准(如「相比某内部版本」)会反向泄露被隐藏事物的存在,
  改为不指向任何具体对象的行业通用参照。
```

### 1.4 overclaim 透镜

```
专用指令:
- 绝对化模式先扫一遍作候选(不是判据,是线索):绝不|从不|不依赖任何|100%|完全不|业界领先|唯一。
- 每个绝对化否定句检查:是否锁死了同文档/同套交付物里刻意留开的备选方案。
- 每个卖点问一句:「客户拿到手实测一次,这句话还站得住吗?」——
  写进方案的第三方平台能力(审批流/压缩/导出)与免费配额必须有实测或官方文档背书。
- 防御性空话(「经业界与学术验证」类)重复出现≥2 次即报,证据留一处即可。
```

### 1.5 时效透镜(易过期硬事实)

```
专用指令:
- 专扫一轮「会过期的事实」:模型/硬件型号、版本代次、实例规格、单价、配额、法规标准版本号。
- 逐项联网核实当前最新;过时项最容易在看起来专业的方案里蒙混过关。
- 建议措辞代际无关化(只写系列名不写具体代次),并评估替换对成本测算是否有连带影响。
```

### 1.6 证据存在性核验(多票对抗)

```
专用指令(对每条关键引用派 N≥3 个独立验证代理):
- 各自独立尝试定位来源原文(检索标题/作者/DOI/官方页面),互不通气。
- 全票无法核实存在 → 该来源整体剔除,即使它完美支持结论、承载多条主张。
- 剔除后检查结论是否仍由其余独立证据链支撑;交付物末尾加透明度声明,
  说明结论不依赖被否决来源。
- 心法:完美贴合结论的证据最可疑,优先核它。
```

## 2. 财务表透镜

### 2.1 算术配平透镜

```
专用指令(独立重算,不看原表公式,只看呈现出来的数字):
- 每个总额 = 分项之和(教学例:分项 30+30+40,总额必须=100);
- 每个差额 = 两方相减(例:方案甲 200 - 方案乙 150 = 差额必须写 50);
- 每个中点 = (低+高)/2;
- 章节间对账:同一数字在汇总表/分项表/正文引用三处一致;
- 范围口径:低~高区间的两端各自可加总闭合,不混入第三种口径。
- 全部清零才允许 commit;任何一处不平,输出「期望值 vs 实际值 vs 位置」。
```

### 2.2 级联完整性透镜

```
专用指令:
- 输入:本轮变更的「旧值→新值」清单(如 100→120、10 周→8 周)。
- 清扫范围按「文档类别」全量枚举(例:全部客户向文件),不是只查任务提及的文件;
  包括生成器源码里的封面文案、文件名、图表标注。
- 用 grep -c 对每个旧值计数,判据=0;发现一处泄漏立即对同类全部文件复扫。
- 有意保留的旧值(内部文档留作依据)列入 clean_declaration 并说明理由。
```

## 3. 视觉透镜

### 3.1 逐页目检透镜

```
专用指令:
- 亲自打开渲染产物(截图/Read 图片),不是确认命令退出码为 0。
- 按版面类型抽样:封面/正文/表格页/嵌图页/多语言混排页各取代表页。
- 查:破版、掉字(尤其 CJK)、标点错位、图表与正文引用编号一致、主题色一致。
- 委派子代理产视觉物时给逐页规格,交付后仍要亲自截图核验——子代理自检通过不算数。
```

### 3.2 原始证据对照透镜(综合稿专用)

```
专用指令:
- 综合稿里每条视觉断言(颜色值/布局/组件状态/参考图文件名)逐条回到原始截图:
  重新打开图片、重新像素取样,不引用上游 agent 的转述。
- 实战教训:某综合稿含 4 处二手臆造(顶栏底色、错位的文件名、凭空色值、按钮颜色猜错),
  全部是「转述的转述」,重读原图才揪出。
- 校正后输出错误对照表(原稿写了什么 → 实物是什么 → 证据文件)。
```

## 4. 售前红队(三视角,材料带去谈之前)

```
专用指令(三个独立视角并行):
- 客户砍价视角:哪句话给了压价把柄?哪个数字经不起追问?
- 合规/过度承诺视角:哪些承诺超出可交付范围?哪些时间点写死了?
- 第三方能力验真视角:方案依赖的每项平台能力逐项实测或查证,
  不能凭印象把不同层级的工具并列推荐;免费版配额(次数/存储)是暗坑,专项列出。
- 单人写作会把「听起来对」的卖点当真——红队后台并行跑,期间先把最可用片段给用户,结果回来折进定稿。
```

## 5. 裁定(adjudication)模板

```
【角色】你是裁定人,收敛各透镜 findings,产出唯一修复清单。
【输入】各透镜 JSON + 基线常量区块(同上)。
【裁定规则】
1. 多透镜共指同一处 → 基本必为真问题,置顶。
2. 与基线常量/用户已明确事实冲突的 finding → 不改稿,降级为「现场核实项」置顶提醒。
3. 机械可检问题(算术/超载/舍入)→ 退回执行者自修,不占评审带宽。
4. 评审期用户可选「只记录不改」→ 挂待办+写 memory,交付物一字不动。
5. 每条修复落地后重新跑对应透镜验证,不以「已修」自述为准。
【输出】修复清单(ASCII 键 JSON)+ 现场核实项 + 有意保留项(含理由)。
```
