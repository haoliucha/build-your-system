# 反模式→正确做法全目录

全部来自某 To-B 投标/交付物项目的实战翻车,已脱敏;示例数字为教学值。按「反模式 → 翻车实录 → 正确做法」组织。标 ★ 的为高权重教训(必看)。

## A. 证据与断言

### ★ A1. 完美证据不核就用
- **反模式**:一条引用完美贴合核心结论,顺手写进客户报告。
- **翻车实录**:一篇看似完美支撑核心技术路线、承载 5 条主张的论文,在 3 票独立对抗验证中被全票否决——无法核实其存在,险些进入客户交付物。
- **正确做法**:完美贴合结论的引用**优先**核存在性;被否决来源整体剔除(全部主张连坐),结论改由多条独立证据链支撑,报告末尾加透明度声明。单一来源被证伪后结论不得坍塌。

### ★ A2. 竞品内幕凭体验写成事实
- **反模式**:凭产品体验语气推断竞品技术形态(「就是通用模型套壳」「对话要过第三方云」),写成方案里的事实断言。
- **翻车实录**:用户质问「这句话有根据么?」,联网取证判为无据;客户给竞品打一通电话即可证伪,连累整个自研叙事的可信度。
- **正确做法**:点名竞品的技术内幕断言必须有公开披露证据;查无披露写「官方未公开,不评判其技术路线」;行业概括类弱证据可留但加限定词;用户指示「不知道就不要提」时彻底删除。

### ★ A3. 平台能力/卖点凭印象承诺
- **反模式**:把「听起来对」的第三方平台能力写成主推卖点;把不同层级的工具并列为等价选项。
- **翻车实录**:主推档核心卖点「平台自动压缩附件」被红队三视角同时证伪——平台默认存原件,「客户一实测导出还是原大小,主推档当场崩」;另把无审批引擎的工具与真审批平台并列三选一;还漏了免费版「每月自动化次数上限+存储上限」两个暗坑。红队共揪出 25 处问题、3 个必改硬伤。
- **正确做法**:写进方案的平台能力与配额逐项实测或查证;卖点改为可验真的机制表述;真需要的能力做成自己交付的一环而非假定平台代劳;配额坑写进风险边界并给绕行方案。

### ★ A4. 易过期硬事实不核时效
- **反模式**:沿用记忆里的型号/版本/价格,方案看起来专业就蒙混过关。
- **翻车实录**:方案用了落后约一代的模型型号(且新代际根本没有该参数档),用户直觉质疑「是最新的么?」并被证实。
- **正确做法**:交付前专扫一轮「会过期的事实」并联网核实;措辞尽量代际无关(只写系列名);替换后验证对成本测算的连带影响(同档结论是否不变),grep 残留=0。

### A5. 查不到就编 / 被墙就绕着编
- **正确做法**:显式列「查不到/需确认」清单+官方核验路径;抓取被登录墙/风控拦截时停下报告或改用已验证的既有材料,绝不围绕抓不到的内容编造。(适用条件:外部网页抓取,尤其登录墙站点。)

### A6. 网页核价被侧栏污染
- **反模式**:同一规格出现两个相近价格,照单全收。
- **正确做法**:在 DOM 容器级别剥离正文与侧栏推荐/相关文章 feed——两个相近价往往是不同文章的数据被拼进同一渲染结果,张冠李戴会把别的规格的折扣条件拼进报价。(适用条件:多源比价抓取。)

## B. 改写与口径

### ★ B1. 定位改写后不审校直接交付
- **反模式**:口径/定位类改写自认「只是换措辞」,改完即交付。
- **翻车实录**:一轮改写自身引入 6 处缺陷:绝对化 overclaim、防御空话重复 3 次、量词撞已否决的旧口号、孤儿引用反向泄露、标题脱括注易误读、并列措辞软暗示——全部被 4 个独立透镜并行审校抓出。
- **正确做法**:定位/口径改写完成后必跑多透镜对抗审校(一致性/脱敏/overclaim/准确性);这类改动的真风险是自引缺陷,不是错字。

### ★ B2. 绝对化否定句锁死备选
- **反模式**:「不依赖任何外部服务」式的绝对句显得立场坚定。
- **翻车实录**:该句与同文档另一节把外部 API 列为正式权衡备选直接矛盾,被 3 个透镜同时标记。
- **正确做法**:改陈述具体正向事实(如「全部数据与运算均在客户自有环境完成」),守住定位又不关死权衡之门。

### B3. 删前提留孤儿引用
- **翻车实录**:删除某前提后,「相比内部封闭版」这类比较基准成孤儿引用,反向暗示存在一个被隐藏的内部版本。
- **正确做法**:删前提后全文扫残留指向;改为不指向任何具体对象的行业通用参照。

### B4. 修辞改写偷渡语义变更
- **翻车实录**:复核发现一处合规义务从「无条件必做」被改成「按需」,且未申报,与改写方「一字未动语义」自述冲突。
- **正确做法**:修辞任务中任何义务强度/内容变更单独申报由作者拍板,即使它恰好消除了旧矛盾。

## C. 数字与口径分层

### ★ C1. 锁定口径只改任务提及的文件
- **反模式**:清扫敏感表述/旧数字时,只处理任务里点名的文件。
- **翻车实录**:某口径清扫只覆盖任务提及的两份文档,漏了同为客户向的排期表,后来改别的问题时才顺手发现泄漏。
- **正确做法**:清扫范围按「文档类别」全量执行(所有客户向文件,含生成器封面文案与文件名);发现一处泄漏立即对同类全部 grep 复扫。

### ★ C2. 对客数字与内部区间混用
- **翻车实录**:内部行情中位数与对客已报口径并存,会议讲解一旦混用就与已报数字当场冲突;对客一页纸与内部详表各自演化,客户对比即露馅。
- **正确做法**:已报客户的数字即锁定口径,对外文件一律用它;内部区间/中位数只留内部文件并标注勿外发勿投屏;会前红线清单写明「别口播内部数」。头条数字两套文件按已报口径统一。

### ★ C3. 跨文档 claim 打架
- **翻车实录**:方案正文把调研报告里明确标注「未公开/需销售核实」的事项写成确定性事实,两份自家文档直接打架——最易被客户戳穿的内伤。
- **正确做法**:审校专设 claim 溯源透镜,肯定断言逐条对照其他交付物的 caveat,统一口径。

### C4. 有意分层不申报
- **正确做法**:客户向撤掉的前提(如内部沿用的供应商折扣假设)可在内部文档故意保留作报价依据——这种分层不一致是设计不是 bug,但必须显式报告,是否同步内部由用户拍板(牵动锁定数字)。

### C5. 财务级联后不做独立算术校验
- **正确做法**:跨文件数字级联完成后,commit 前用独立对抗校验重算全部配平关系(总额=Σ分项、差额=相减、中点、章节对账),清零才提交。教学例:分项 30+30+40 则总额必须=100,差额 200-150 必须写 50。

## D. 综合稿与视觉

### ★ D1. 综合稿二手臆造
- **反模式**:多 agent 工作流的综合稿引用上游 agent 的转述,不回原始证据。
- **翻车实录**:综合稿含 4 处臆造:顶栏底色写反(与实测帧不符)、参考图文件名错位、渐变色值凭空给出、按钮颜色猜错;审校 agent 重读已存图片文件才全部揪出。
- **正确做法**:综合稿必过一道**重读原始证据**的对抗审校;交付物只编码取样或直读到的事实;校正后列错误对照表。

### D2. 命令跑通当排版正确
- **正确做法**:视觉产物(图/PDF/PPT)重生成后亲自逐页/按版面类型抽样目检(封面/正文/表格/嵌图/多语言页);委派子代理时给逐页规格,交付后仍亲自截图核验。

### D3. 子代理结论不对实物直接转述
- **正确做法**:后台子代理看不到用户给的截图/文件时,其分析结论转述给用户前先与实物逐数核对,并明说「已核对,数字完全吻合」。

## E. 检查器与验证

### E1. grep 管道假阴性
- **翻车实录**:用 `grep -o … | head` 检查旧值残留,管道掩盖了 grep 退出码,误判「已清零」,产物里旧值仍在。
- **正确做法**:清零判据用 `grep -c` 计数=0;或用本 skill 的 `scripts/check-residuals.sh`。

### E2. 报错的检查当通过
- **翻车实录**:多文件变量未 word-split 被当成单一路径,三项脱敏检查实际是报错而非扫描,险些当成通过。
- **正确做法**:检查命令报错≠扫描通过;用显式文件列表重跑,拿到真实零命中才算数。

### E3. 绿灯检查不验证有效性
- **正确做法**:检查通过后注入一个已知错误,确认检查器真的在查目标文件且能抓到,删除后复测干净——防空转假通过。`check-residuals.sh selftest` 即此模式的固化。

### E4. 宣称根因/修复前不做最小复现
- **正确做法**:先构造最小隔离复现(bad case),同一样本上验证修复(good case),全量重生成前再用含问题特征的冒烟样本试渲染。「产物为什么缺 X」类问题先取证对比正常/异常产物的元数据与生成链路,再下结论。

## F. 审校自身的边界

### F1. 审校意见顶撞用户已明确的事实
- **翻车实录**:红队从样表推断「单人每月约 10 笔」进而要求推翻整个多人方案,但用户此前已明确实际是「多人每月上百条」。
- **正确做法**:按用户事实定锚,冲突转成置顶的现场核实项(开谈先摸底坐实,不符立即降档换案)。

### F2. 被质疑时先辩护再取证
- **正确做法**:用户质疑某论断时先给诚实初判(包括承认站不住),再跑多路联网取证+对抗裁定,产出多档改写口径(保守/带锋芒)由用户定调;用户选「只记录不改」则挂待办+写 memory,交付物一字不动。

### F3. 机械问题占用评审带宽
- **正确做法**:超载/倒挂/舍入溢出等可确定性检测的问题送审前自修;评审只挑判断类问题——评审是校验,不是替执行者做平衡。
