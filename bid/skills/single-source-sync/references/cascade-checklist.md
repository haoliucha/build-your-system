# 口径级联与手改回捕 — 详单

SKILL.md 的展开版。所有案例抽象自某 To-B 投标项目的实战复盘,示例数字为教学值。

## 一、口径级联固定链(七步展开)

**第 0 步 · 前提检查**。动手前判断指令性质:

- 若会**反转用户已锁定的决策**或使头条报价数字大幅移动(如某档计 0 让总价低端掉两成多):先算出级联后的完整数字表呈给用户,确认「这是你要的吗」与同步范围(全量/部分),再执行。
- 若用户的问题**实质推翻了此前锁定的前提**(部署模型/租户模型/规模档):点破「这个岔路口会改变所有下游数字」,让用户明确拍板后一次性重写受影响章节,不在旧前提上缝补数字。
- 写新一轮交付物前,先读回此前已验证的调研结论/事实底座,保证全套材料口径同源不漂移。

**第 1 步 · 映射爆炸半径**。全库 grep 旧值/旧措辞,把命中点分两类列清单:

- 公式驱动(生成器参数派生)→ 改源后自动级联;
- 硬编码副本 → 逐处手改清单。

易漏的级联对象(每一类都出过残留事故):

| 藏身处 | 说明 |
|---|---|
| 构建脚本内的模板文案 | PDF 封面/目录常模板化在脚本里,不在 md 中,历次级联最易漏 |
| 兄弟文档 | 同一主张散布在正文十余处 + 兄弟客户向文档 |
| memory/纪要 | 记录口径数字的辅助文件同属级联范围,提交前核对与同批交付物一致 |
| console 标签/备注 | 非正文位置的旧措辞 |
| 计数点 | 增删案例/章节后,正文总数、目录、附录索引、方法说明全部要同步(顺带核 off-by-one) |

同时识别**只有定性表述**的文档可豁免重生成,精确圈定半径,避免无谓全量重建。

**第 2 步 · 改源**:只改数据文件/生成器参数,不直接动产物。

**第 3 步 · 重跑生成器**,核对 console 关键数字与预期一致。「只改呈现不改口径」的重构以「全部关键数字与改前逐一相同」为验收线。

**第 4 步 · 同步叙事文档**。批量 replace 后必须重读每个替换点的行文语境:

- 反模式:blunt replace_all 留下「占高」类病句;第一次修补又把明令废弃的旧标签改了回去。
- 双规范:表格用裸短标签,散文用完整措辞;逐处润色后再终验。
- 同词多义要区分:清除某个词的问题语义时,保留其合法用法——如清「平台迁移」一词时,别误伤「数据迁移」「迁移测试」,同词多义逐条判读不连坐。
- 章节删除连带重排全部交叉引用,并区分「章节引用」与「法条/标准条款引用」,后者不可误改。

**第 5 步 · 重生成受影响产物**(xlsx/PDF/HTML/图),按第三节抽验。

**第 6 步 · grep 残留清零**。对文档 + 生成器 + 生成 HTML 全树 grep 旧关键词直至清零,命中逐条判读。合法误报清单(不算残留,但要逐条写明理由):

- 数字子串:365 含 65、1010 含 10;
- 改动说明/规则自述:描述「旧值 → 新值」的文字本身;
- 合法范围值:区间端点恰与旧值同数;
- 同词异义的合法用法。

**第 7 步 · memory 更新 + 分组提交**(见第五节)。收尾报告:实际改动集与计划一致(git diff 核对),并显式声明「验证过未动」的文件清单。

## 二、手改回捕 SOP(xlsx)

触发:用户直接在办公软件里改了生成的表格(改参数、改数字、删行/删列),要求「捕捉我改了什么 / commit / 级联更新」。

1. **备份**:`cp 手改版.xlsx 手改版.xlsx.bak`。
2. **重生成对照版**:从当前源跑生成器得到「干净版」。若担心覆盖,可先 regen 到临时目录再比对,最后把手改版还原回原位并校验与备份 byte-identical(证明比对过程零破坏)。
3. **逻辑值 diff**:`xlsx-dump.cjs` 分别 dump 两版后 diff。差异行 = 手改意图。办公软件保存会重排内部 XML,raw 字节 diff 噪声极大,不可用。
4. **意图判读**:逐条区分「改参数值 / 改口径 / 删行删列 / 误触」。删除类先向用户确认是有意还是误删。手改常自相矛盾——主表改全了,但汇总曲线、结论行、颜色标注、备注漏改;指出矛盾点,**拒绝 commit 矛盾状态**。
5. **落源**:把确认的意图写进生成器(参数默认值更新、条目删除、列删除),保证「改参数格仍自动重算」。
6. **干净重生成** + 抽验(第三节),漏网的派生字段在此一并修复。
7. **一次 commit**:提交源 + 产物,说明中注明「手改已回捕落源」。原始手改文件留 .bak 并告知用户。

**前置铁律 — lsof 查写句柄**(macOS;WPS/Excel 等桌面办公软件场景):重生成任何可能被打开的产物前,`lsof <file.xlsx>` 检测;检出即**停下**让用户「关闭不保存」再生成,并提醒重开刷新。否则软件会把内存旧副本回存,覆盖脚本刚写出的新文件——实测同一项目复发 3 次。用户报「内容回退到旧版」时,用 mtime + 磁盘内容诊断,多半就是回存覆盖;源完好则直接重生成即可。

## 三、重生成抽验清单

命令零退出 ≠ 产物已更新(管线某步可能静默失败,曾有 PDF 加书签步骤静默失败只打印一个裸括号,产出陈旧 PDF)。每次重生成后:

- [ ] 新口径字符串已出现(xlsx 单元格实值 / HTML 关键串 / PDF 页内容);
- [ ] 旧口径字符串已消失(对编译产物也 grep);
- [ ] 单元格 numFmt / 小数位 / 页数符合预期;
- [ ] xlsx 是否真更新以逻辑值 dump 为准,不看 mtime/git modified(xlsx 内嵌时间戳,字节级非确定);
- [ ] 确定性渲染的图类:未涉及产物字节零变化 = 范围守恒证据;
- [ ] 用户报「文件没更新」:先核实磁盘实态再下结论;澄清表象后继续深挖,背后常藏另一个真 bug;
- [ ] 派生简版/面向非技术读者的清单:逐数字核对与主模型一致;
- [ ] 小计正确 ≠ 明细行同步:对逐行派生值做对抗式复核。

## 四、GUI 图形文件(drawio 类)回捕

用户在 GUI 工具里手工调排版时,与 xlsx 同理但用**几何/结构 diff**:GUI 保存会重排 XML 序列化顺序,全部页面都有字节差异;解析出坐标/尺寸/文本做语义比对,定位真实改动,逐项 port 回代码 spec 重生成。不要直接拿手改文件转 PNG——会丢生成管线独有的富元素(徽标/备注层)。

## 五、提交纪律细则

- 显式列文件路径暂存,禁 `git add -A`;先 dry-run pathspec 确认不误扫,commit 前列暂存集逐一核对「暂存 = 本任务 N 个文件,无关改动全在外」。
- 无关预存改动(他人遗留/字节噪声/参考项目产物被构建误触)不动不提交,点名报告;误触出 scope 的先 `git restore` 还原。用户自建或来源不明的文件只提示,不代提交。
- 按交付物拆分组 commit;文件改名产生的删除易被并入前一组,导致后组 stale pathspec 失败——分组后逐组核对。
- 字面「提交所有」默认仍走排除纪律只提任务文件;仅当用户在被明确告知排除方案后**强调式重申**,才视为知情覆盖、全量提交,并在回报中注明。
- 运行会写仓的外部工具(双向同步类)前,先提交干净基线,工具改动即成可审可回退的 diff。
- 泛化既有构建工具服务新产物:新建 CLI 参数化版本,不改已支撑既有交付物的旧脚本,防回归已提交产线。
- 一次性摸索出的工序(采集/样式/生成)沉淀为仓库内可复用脚本 + 配置,记入项目 memory。

## 六、对客场景加固

- **演示/投屏前**:从生成器重出全部表格,交叉核对各文件头条数字一致、无 #REF!、金额不低于自己标注的行情区间下限。用户手改过的文件是最高风险源(曾有会前手改两份表只改一半、当晚投屏必穿帮的险情)。
- **会前口径桥**:主动枚举客户可能抓到的跨文档不一致点(工期表述/金额算例/等级口径),每条备好解释话术,不赌客户看不见。
- **口号与新事实冲突**:口径转向后,原有营销口号若与新事实矛盾,最低限度在一处定义点重新界定含义兜底,并把「是否全面改写」的选择权显式上报用户,不静默保留矛盾话术。
- **评审发现跨文档矛盾**(同一义务两处标注不同强制级别):对齐到统一口径并立即重出所有受影响衍生物。
- **向用户讲解数字**:先用生成器/代码把每一步实跑出精确值核验吻合,再逐行拆公式讲假设来源,不凭记忆复述;顺带标明哪些是硬数据、哪些是可替换假设。
